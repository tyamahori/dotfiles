# task-briefing

ざっくりしたタスクを、チェックリストの6観点（Why・課題・ゴール・スコープ・制約・成果物）が埋まったブリーフィングに対話で仕上げる。全部を質問攻めで埋めるのではなく、**リポジトリと文脈から推定できるものは自分で埋め、本人にしか答えられない不足だけを聞く**。

チェックリストの正本は `~/dotfiles/agents/task-briefing.md`。観点の定義・良い例悪い例・テンプレートはすべてそちらにあり、このファイルには複製しない。

## 手順

### 1. チェックリストを読む

`~/dotfiles/agents/task-briefing.md` を読む。ただし Claude Code では UserPromptSubmit フックが毎プロンプトに同ファイルを注入しているので、すでにコンテキストにあれば再読不要。

### 2. タスクの種を受け取る

引数にタスクの説明があればそれを種にする。なければ「どんなタスクか、まずざっくり教えてください」とだけ聞く。この時点で整った形を求めない。

### 3. 埋められる観点を自分で埋める

種とリポジトリの文脈（コード、git log、直近の会話、開いている issue/PR）から、6観点のうち推定できるものを埋める。推定で埋めた項目には（推定）と印を付け、後で確認に回す。

### 4. 不足だけを質問する

埋まらなかった・自信のない観点だけを質問する。優先順位:

1. **必須4項目**（課題・ゴール・Why・成果物の形）— どれかが空か推定のままなら必ず聞く
2. スコープ / 非スコープ — 逸脱リスクが高そうなタスク（リファクタ誘惑のある場所、複数サービスに跨る変更）のときだけ聞く
3. 制約・前提 — 期限や互換性が結果を左右しそうなときだけ聞く

質問は一度に最大4つ。AskUserQuestion が使える環境ではそれを使い、選択肢には推定した答えを第一候補として載せる。使えない環境（Codex 等）では番号付きの質問を平文で出す。

### 5. ブリーフィングを提示して確認を取る

チェックリストのテンプレート形式で完成版を提示する。タスクが小さければ3行版でよい。推定で埋めた項目は（推定）の印を残したまま見せ、「これで着手してよいか」を確認する。

### 6. 合意後

- そのまま着手を求められたら、ブリーフィングを枠組みとしてタスクを開始する。着手時に枠組みを一言で復唱する（global-instructions の Task intake と同じ運用）。
- ブリーフィングだけが目的（別セッションや他のエージェントに渡す等）なら、Markdown をそのまま出力して終える。ファイルへの保存は頼まれたときだけ。

## やらないこと

- 全観点を機械的に質問する尋問。埋まっている・推定できる観点は聞かない。
- 些細なタスク（typo 修正、指示どおりのコマンド実行）への適用。チェックリスト自身が免除している。
- ブリーフィング作成のついでの実装着手。着手はステップ5の確認を通ってから。
