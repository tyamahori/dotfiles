# タスク指示の観点チェックリスト

コーディングエージェント（Claude Code / Codex / Copilot CLI）に指示を出すとき、
書き手（人間）が埋めるべき観点のチェックリスト。エージェント側も
`global-instructions.md` の「Task intake」セクションに従い、着手前に
これらの観点が揃っているかを確認する。

指示の質を決めるのは手段の細かさではなく、**課題・ゴール・why の明確さ**。
手段はエージェントに任せてよいが、この3つが曖昧なまま渡すと、過剰実装・
過少実装・的外れな最適化のどれかに倒れる。

## 観点

### 1. Why / 背景 — なぜ今やるのか

- 何がトリガーか（バグ報告、新要件、技術的負債、調査で見つけた問題…）。
- これが決まると、エージェントの実装中のトレードオフ判断（どこまで丁寧に
  やるか、速度と品質のどちらに寄せるか）が正しくなる。
- 例:「来週のリリースに入れたい応急処置」と「恒久対応」では正解が違う。

### 2. 課題設定 — 解決したい課題は何か

- 「やってほしい作業」ではなく「解決したい課題」で書く。作業を指定すると、
  それが最適でなくてもエージェントはそのまま実行する。
- 悪い例:「◯◯にリトライ処理を足して」
- 良い例:「△△のAPIが不定期に落ちて夜間バッチが止まる。止まらないように
  したい（手段は任せる）」
- 誰が困っているか（自分、チーム、エンドユーザー）も一言あると優先度判断に効く。

### 3. ゴール / 成功条件 — 何をもって完了か

- 完了の定義を**検証可能な形**で書く。「動くように」ではなく
  「Xの操作をしたらYになる」「このテストが通る」。
- 「どう確認したらOKと言えるか」を1〜2行で書ければ十分。書けないなら
  ゴールがまだ曖昧というシグナル。

### 4. スコープ / 非スコープ — やらないことを明示する

- 触ってよい範囲（このディレクトリだけ、このサービスだけ）。
- 明示的にやらなくていいこと（周辺のリファクタ不要、テストは既存パターン
  踏襲でよい、ドキュメント更新は別タスク、など）。
- エージェントのスコープ逸脱の大半は、非スコープの明示で防げる。

### 5. 制約・前提

- 期限・緊急度、互換性要件（壊してはいけないAPI・挙動）。
- 使ってよい / だめなライブラリや手法。
- 参考にすべき既存実装・過去のPR・ドキュメントがあればパスやURLを渡す。

### 6. 成果物の形

- 何が返ってきたら受け取り完了か: draft PR、コミットのみ、調査レポート、
  設計提案（コード変更なし）、など。
- レビューだけ頼みたいのか、修正までしてほしいのかもここで区別する。

## テンプレート

コピペして埋める。該当なしの項目は消してよい。

```
## 背景 / Why
（なぜ今やるのか。トリガーは何か）

## 課題
（解決したい課題。作業指定ではなく課題で書く。困っているのは誰か）

## ゴール / 成功条件
（完了時の状態。どう確認したらOKと言えるか）

## スコープ / 非スコープ
- やる:
- やらない:

## 制約・前提
（期限、互換性、ライブラリ、参照すべき既存実装）

## 成果物
（draft PR / コミット / 調査レポート / 設計提案 …）
```

### 3行版（小さいタスク用）

```
課題: （何に困っているか）
ゴール: （どうなったら完了か・確認方法）
補足: （スコープ外・制約があれば）
```
